## 前言：为什么用毛主席思想学算法

刷过力扣的人都有这种体验：看答案都会，关上答案就废。为什么？因为你在**背模板**，而不是在**分析矛盾**。

毛主席在《矛盾论》中写道："事物发展的根本原因，不是在事物的外部而是在事物的内部，在于事物内部的矛盾性。"算法题也一样——每道题都有一个**核心矛盾**：暴力能做但太慢、空间不够、找不到规律……你只要找到了这个矛盾，解法自然浮现。

举一个最简单的例子。两数之和（Two Sum）：给定一个数组和一个目标值，找两个数使它们之和等于目标值。暴力做法是两层循环枚举所有数对——O(n²)。矛盾是什么？**查找补数太慢**。你每次都要重新扫描整个数组去找 `target - nums[i]` 是否存在。矛盾找到了，解法就有了：既然查找是瓶颈，那就用一个查找快的数据结构——哈希表，O(1) 查补数。

看，你不是在"套模板"，你是在"解决矛盾"。这就是红宝书的全部方法。

本电子书按以下结构组织：第一章讲矛盾论的几个核心概念，第二章把每个数据结构拆解为"矛盾 → 语录 → 模板 → 例题 → 易错"，第三章覆盖五大核心算法，第四章给出持久战学习路线。每一章都贯彻同一个原则：**先找矛盾，再选武器**。

## 第一章：矛盾论基础

### 1.1 什么是矛盾分析法

> "一切事物中包含的矛盾方面的相互依赖和相互斗争，决定一切事物的生命，推动一切事物的发展。"
>
> ——《矛盾论》

**矛盾分析法**就是在面对一个问题时，先不急着找答案，而是问自己三个问题：

1. 已知条件是什么？（输入、约束）
2. 所求结果是什么？（输出）
3. 已知和所求之间的差距是什么？（**这就是矛盾**）

在算法题中，矛盾通常表现为：暴力解法能做，但**效率不够**；或者暴力根本做不了，因为**规模太大**。

**调查三步走**（每次做题必须执行）：

```
1. 输入：数据类型、数据范围、是否有特殊值
2. 输出：返回值类型、格式要求
3. 约束：时间限制暗示什么复杂度？空间限制暗示什么？
```

"没有调查就没有发言权"——没搞清楚这三条就写代码，等于瞎指挥。

### 1.2 主要矛盾与次要矛盾

> "在复杂的事物的发展过程中，有许多的矛盾存在，其中必有一种是主要的矛盾，由于它的存在和发展规定或影响着其他矛盾的存在和发展。"
>
> ——《矛盾论》

一道算法题里，矛盾不止一个，但**只有一个是主要的**。主要矛盾解决了，次要矛盾（边界条件、代码细节）迎刃而解。反过来，如果你盯着次要矛盾（比如纠结变量命名），永远解不了题。

**判断主要矛盾的清单**：

- 暴力为什么不行？→ 时间复杂度是主要矛盾 → 换算法
- 暴力能过但逻辑不对？→ 逻辑是主要矛盾 → 重新分析
- 逻辑对但边界挂了？→ 边界是主要矛盾 → 补边界
- 全对但超时？→ 效率（常数级优化）是主要矛盾 → 剪枝

**诊断代码时的四层顺序**（不可颠倒）：

| 层级 | 诊断内容 | 语录 |
|------|---------|------|
| 1. 逻辑矛盾 | 思路根本不对 | "路线错了，知识再多也没用" |
| 2. 边界矛盾 | 空、单、极值挂了 | "没有调查就没有发言权" |
| 3. 工具矛盾 | 数据结构选错 | "不同质的矛盾用不同质的方法解决" |
| 4. 效率矛盾 | 常数太大 | "在战术上重视敌人" |

一次只诊断一层，确认修正后再进入下一层。不要一次性指出所有问题——那是主观臆断，不是调查研究。

### 1.3 矛盾的普遍性与特殊性

> "矛盾的普遍性即寓于矛盾的特殊性之中。"
>
> ——《矛盾论》

**普遍性**：每道算法题都有矛盾。不是这道题特别难，是所有题都需要分析矛盾。

**特殊性**：每道题的矛盾**具体是什么**，各不相同。两数之和的矛盾是"查找太慢"，爬楼梯的矛盾是"重复计算太多"，接雨水的矛盾是"每个位置上能接多少水由旁边的柱子决定"——矛盾不同，解法不同。

这就是为什么**不能背模板**。背模板 = 忽视特殊性 = 形式主义。你必须在每道题上重新调查、重新分析矛盾。但同时，你会逐渐发现：某些类型的矛盾经常出现，解法也就形成了几种范式（哈希、双指针、DP……），这叫做**从感性认识到理性认识的飞跃**。

### 1.4 语录速查表

| 主席语录 | 适用场景 | 一句话解释 |
|---------|---------|-----------|
| 没有调查就没有发言权 | 审题阶段 | 搞清楚输入输出约束再动手 |
| 集中优势兵力，各个击破 | 二分/双指针 | 每次集中力量消灭一半 |
| 牵一发而动全身 | 链表 | 改动一个指针影响全局 |
| 人民群众是历史的创造者 | 哈希表 | 每个元素都是群众，发动群众帮你查 |
| 前途是光明的，道路是曲折的 | 栈与队列 | 后进先出/先进先出，绕路但终达目标 |
| 抓住主要矛盾 | 二叉树递归 | 抓住 root，子树同理 |
| 星星之火，可以燎原 | 动态规划 | dp[0] 是星火，dp[n] 是燎原 |
| 敌进我退，敌退我进 | 回溯 | 前进做选择，后退撤销选择 |
| 当机立断 | 贪心 | 每一步做当前最优选择 |
| 联系与发展 | 图论 | 节点之间相互联系，状态不断发展 |

## 第二章：数据结构篇

### 2.1 数组——集中优势兵力，各个击破

> "集中优势兵力，各个击破，是战胜敌人的根本法则。"

**核心矛盾**：数组是顺序存储的线性结构。随机访问 O(1)，但查找需要遍历 O(n)。当题目要求在线性结构上做查找、区间操作、或原地修改时，矛盾就是**如何避免 O(n²) 的暴力扫描**。

**武器库**：
- **双指针**（左右夹击）：左右两端各放一个指针，根据条件向中间收缩。"集中兵力"于两端，不是分散遍历全部。
- **滑动窗口**（动态收缩）：维护一个可变大小的窗口，右端扩展、左端收缩。"集中兵力"于窗口内的元素。

**模板：双指针（两端向中间）**

```python
left, right = 0, len(nums) - 1
while left < right:
    cur = nums[left] + nums[right]  # 或其他运算
    if cur == target:
        return [left, right]
    elif cur < target:
        left += 1   # 集中兵力，消灭左端
    else:
        right -= 1  # 集中兵力，消灭右端
```

**模板：滑动窗口**

```python
left = 0
window_state = 0  # 窗口内状态
for right in range(len(nums)):
    window_state += nums[right]  # 扩展右端
    while window_state > target:  # 矛盾：窗口超过目标
        window_state -= nums[left]  # 收缩左端
        left += 1
    # 此时窗口满足条件，处理
```

**经典例题**：
- [三数之和](https://leetcode.cn/problems/3sum/) —— 排序 + 固定一个 + 双指针夹逼。矛盾：三层循环 O(n³) → 排序后双指针 O(n²)。
- [盛最多水的容器](https://leetcode.cn/problems/container-with-most-water/) —— 双指针从两端向中间，每次移动较矮的一边。
- [无重复字符的最长子串](https://leetcode.cn/problems/longest-substring-without-repeating-characters/) —— 滑动窗口 + 哈希集记录窗口内字符。

**常见错误**：
1. 双指针移动条件写反——该移左边还是右边，取决于**目标值的性质**，不是随机选。
2. 滑动窗口的 while 条件写成 if——当窗口需要持续收缩时，if 只收缩一步，会导致窗口状态不合法。
3. 忘记排序——双指针（夹逼法）要求数组有序，否则指针移动无意义。

### 2.2 链表——牵一发而动全身

> "牵一发而动全身。"

**核心矛盾**：链表是物理上分散存储、通过指针串联的线性结构。和数组相反：插入删除 O(1)，但随机访问 O(n)。主要矛盾是**修改一个节点的 next 指针，整条链的结构就变了**——这正是"牵一发而动全身"。

**武器库**：
- **虚拟头节点（dummy）**：消除头节点的特殊性——没有 dummy 时，删除头节点和删除中间节点的代码不一样。加了 dummy 后，头节点和中节点统一处理。"矛盾的普遍性"——dummy 把特殊情况统一为一般情况。
- **快慢指针**：利用两个指针的速度差。快指针走两步，慢指针走一步。相遇 → 有环；快指针到末尾 → 中点。"利用矛盾双方的不平衡性"。
- **反转链表**：prev、cur、nxt 三指针，逐个扭转方向。每次只改变一个节点的指向——局部改变影响全局。

**模板：反转链表**

```python
prev, cur = None, head
while cur:
    nxt = cur.next   # 保存后路
    cur.next = prev  # 扭转方向——牵一发
    prev = cur       # 向前推进
    cur = nxt        # 向前推进
return prev          # 整条链已被反转——动全身
```

**模板：快慢指针找中点**

```python
slow = fast = head
while fast and fast.next:
    slow = slow.next       # 走一步
    fast = fast.next.next  # 走两步
# slow 在中点（奇数个）或右中点（偶数个）
```

**经典例题**：
- [反转链表](https://leetcode.cn/problems/reverse-linked-list/) —— prev/cur/nxt 三指针，迭代扭转。
- [环形链表](https://leetcode.cn/problems/linked-list-cycle/) —— 快慢指针，有环必相遇。
- [合并两个有序链表](https://leetcode.cn/problems/merge-two-sorted-lists/) —— dummy + 双指针比较，谁小接谁。

**常见错误**：
1. 反转链表时忘记保存 `cur.next`，导致链表断裂找不到后续节点。
2. 快慢指针的 while 条件写成 `while fast.next`——当 fast 走到最后一个节点时，`fast.next.next` 会空指针。
3. 删除节点时没有用 dummy——删除头节点的逻辑和删除中间节点不一样，忘记处理头节点导致 bug。

### 2.3 哈希表——人民群众是历史的创造者

> "人民群众是历史的创造者。"

**核心矛盾**：查找太慢。数组里找一个元素需要 O(n) 遍历。能不能让每个元素自己"站出来"告诉我们它在哪？

**原理**：把每个元素当作"群众"，存入哈希表后，发动群众帮你查找——O(1) 时间就能找到任何一个元素。空间换时间，用"群众的力量"（额外的存储空间）换取"历史的飞跃"（效率提升）。

**模板：哈希表辅助查找**

```python
seen = {}  # 发动群众——创建群众组织
for i, num in enumerate(nums):
    complement = target - num   # 需要找谁
    if complement in seen:      # 群众里有他吗？
        return [seen[complement], i]  # 有！立刻响应
    seen[num] = i               # 登记入册——发展新群众
```

**经典例题**：
- [两数之和](https://leetcode.cn/problems/two-sum/) —— 哈希表存补数，O(n) 一遍过。
- [存在重复元素](https://leetcode.cn/problems/contains-duplicate/) —— 哈希集记录见过没。
- [字母异位词分组](https://leetcode.cn/problems/group-anagrams/) —— 排序后的字符串做 key，发动群众分类。

**常见错误**：
1. 先建表再查找——应该边遍历边查边存，一次遍历完成。
2. 忘记处理哈希冲突——Python 的 dict 已内置处理，但概念上要知道：两个不同 key 算出了同一个 hash 值，会碰撞。
3. 用哈希表存了不必要的信息——只用判断存在性时，用 set 而不是 dict，省空间。

### 2.4 栈与队列——道路是曲折的，前途是光明的

> "前途是光明的，道路是曲折的。"

**核心矛盾**：处理顺序的两种对立需求。有时候需要"后来的先处理"（栈，LIFO），有时候需要"先来的先处理"（队列，FIFO）。不管哪种，最终都能完成任务——"道路是曲折的，前途是光明的"。

**武器库**：
- **栈（Stack）**：后进先出。适用场景：括号匹配、函数调用、表达式求值、单调栈找下一个更大元素。
- **队列（Queue）**：先进先出。适用场景：BFS 层序遍历、任务调度、滑动窗口最大值（双端队列）。
- **单调栈（Monotonic Stack）**：栈内元素单调递增或递减。每次新元素入栈前，先把破坏单调性的元素弹出——"清除挡路的障碍"。

**模板：单调栈（找下一个更大元素）**

```python
stack = []  # 单调递减栈，存下标
result = [-1] * len(nums)
for i, num in enumerate(nums):
    while stack and num > nums[stack[-1]]:
        # 新来的更大 → 前面小的该让位了
        result[stack.pop()] = num  # 道路曲折：小的被弹出去
    stack.append(i)  # 前途光明：大的留下来了
```

**经典例题**：
- [有效的括号](https://leetcode.cn/problems/valid-parentheses/) —— 栈匹配括号，左括号入栈，右括号出栈配对。
- [每日温度](https://leetcode.cn/problems/daily-temperatures/) —— 单调栈找下一个更高温度。
- [用栈实现队列](https://leetcode.cn/problems/implement-queue-using-stacks/) —— 两个栈倒来倒去，曲折但实现队列功能。

**常见错误**：
1. 单调栈里存的是下标还是值——根据题目定，存下标可以同时拿到值和位置。
2. 栈空时直接 pop——任何时候 pop 先检查 `if stack`。
3. BFS 用栈实现——BFS 必须用队列（先进先出），用栈会变成 DFS。

### 2.5 二叉树——抓住主要矛盾

> "抓住主要矛盾。"

**核心矛盾**：树是递归定义的——一棵树由一个根节点和多棵子树组成，每棵子树又是一棵树。代码不能自己调用自己……等等，可以！递归就是让代码自己调用自己。

**原理**：在任何一个节点上，你只需要关心**当前节点**该做什么，把子树的事情交给递归去做。这就是"抓住主要矛盾"——当前节点是主要矛盾，子树是次要矛盾，递归自动帮你处理次要矛盾。

**模板：二叉树遍历（递归）**

```python
def traverse(root):
    if not root:
        return
    # 前序位置：刚进入当前节点，子树还没处理
    # → 自顶向下传信息（如计算深度）
    traverse(root.left)
    # 中序位置：左子树处理完，右子树还没处理
    # → BST 的中序是有序的
    traverse(root.right)
    # 后序位置：左右子树都已处理完
    # → 自底向上汇总信息（如计算节点数）
```

**经典例题**：
- [二叉树的最大深度](https://leetcode.cn/problems/maximum-depth-of-binary-tree/) —— 后序：`1 + max(左深, 右深)`。
- [翻转二叉树](https://leetcode.cn/problems/invert-binary-tree/) —— 前序：交换左右子树，递归翻转。
- [验证二叉搜索树](https://leetcode.cn/problems/validate-binary-search-tree/) —— 中序遍历结果是否严格递增。

**常见错误**：
1. 前序、中序、后序分不清——关键看"处理当前节点的时机"在递归调用的前面、中间还是后面。
2. 递归忘记 base case——`if not root: return` 是必须的，否则无限递归。
3. 误以为 BST 的中序遍历一定给你一个有序列表——如果 BST 定义不严格（允许等于），需要特殊处理。

## 第三章：算法篇

### 3.1 二分查找——集中优势兵力，各个击破

> "集中优势兵力，各个击破。"

**核心矛盾**：有序数组里找一个目标值。线性扫描太慢（O(n)），但数组的有序性没有被利用。**如何利用有序性快速定位？**

**原理**：每次不检查所有元素，而是集中火力检查**中点**。根据中点与目标值的大小关系，砍掉不可能的一半。每次砍半，O(log n)——"各个击破"。

**模板：标准二分查找**

```python
left, right = 0, len(nums) - 1
while left <= right:
    mid = left + (right - left) // 2  # 防溢出
    if nums[mid] == target:
        return mid            # 找到了
    elif nums[mid] < target:
        left = mid + 1        # 消灭左半边
    else:
        right = mid - 1       # 消灭右半边
return -1
```

**变体**：找左边界（第一个等于 target 的位置）、找右边界（最后一个等于 target 的位置）。核心区别在于 `nums[mid] == target` 时的处理——找左边界时收缩右边界，找右边界时扩张左边界。

**经典例题**：
- [二分查找](https://leetcode.cn/problems/binary-search/) —— 标准模板。
- [搜索插入位置](https://leetcode.cn/problems/search-insert-position/) —— 找不到时返回 left。
- [在排序数组中查找元素的第一个和最后一个位置](https://leetcode.cn/problems/find-first-and-last-position-of-element-in-sorted-array/) —— 左右边界二分。

**常见错误**：
1. `mid = (left + right) // 2` 在 left + right 很大时会溢出（Python 不会，但习惯性用 `left + (right - left) // 2`）。
2. while 条件是 `<=` 还是 `<`——和 right 的初始值及区间定义有关。`[left, right]` 闭区间用 `<=`。
3. 更新时写 `right = mid` 而不是 `right = mid - 1`——会导致死循环。

### 3.2 回溯算法——敌进我退，敌退我进

> "敌进我退，敌退我进。"

**核心矛盾**：需要穷举所有可能的组合/排列/子集，但穷举空间太大。不能无脑 for 循环。**如何在穷举过程中动态前进和后退？**

**原理**：回溯 = 深度优先搜索 + 状态撤销。在决策树的每个节点上，"前进"是做选择（进入下一层），"后退"是撤销选择（回到上一层）。前进和后退互相依存——没有后退，就无法探索其他分支；没有前进，就永远在原地。

**模板：回溯框架**

```python
def backtrack(路径, 选择列表):
    if 满足结束条件:
        result.append(路径[:])  # 记录结果
        return
    for 选择 in 选择列表:
        if 选择不合法:
            continue
        做选择                          # 敌进我退的"进"
        路径.append(选择)
        backtrack(路径, 新选择列表)
        路径.pop()                     # 敌进我退的"退"
        撤销选择
```

**经典例题**：
- [全排列](https://leetcode.cn/problems/permutations/) —— 标准回溯，用 visited 标记已选。
- [子集](https://leetcode.cn/problems/subsets/) —— 回溯，每个元素二元选择：选或不选。
- [N 皇后](https://leetcode.cn/problems/n-queens/) —— 回溯 + 剪枝，列/对角线冲突检测。

**常见错误**：
1. 忘记撤销选择（pop/恢复状态）——导致状态污染，影响其他分支。
2. 结果添加时没有拷贝（`路径[:]`）——直接传引用会导致后续修改污染已存结果。
3. 剪枝条件写错——多剪了漏解，少剪了超时。

### 3.3 动态规划——星星之火，可以燎原

> "星星之火，可以燎原。"

**核心矛盾**：要求全局最优解，但全局问题太大无法直接求解。**如何把一个"燎原大火"拆成无数个"星星之火"，让它们逐步蔓延？**

**原理**：动态规划的哲学是量变引起质变。`dp[0]` 是一块根据地（星星之火），`dp[1]` 在 `dp[0]` 的基础上扩展（火势蔓延），……最终 `dp[n]` 是全局最优（燎原大火）。关键是找到**状态转移方程**——火势如何从 i-1 蔓延到 i。

**模板：DP 三要素**

```python
# 1. 定义 dp 数组含义（根据地）
dp = [0] * n
dp[0] = base_case  # 星星之火

# 2. 状态转移（火势蔓延）
for i in range(1, n):
    dp[i] = max(dp[i-1] + nums[i], nums[i])  # 或者 min、sum 等

# 3. 返回结果（燎原）
return max(dp)
```

**经典例题**：
- [爬楼梯](https://leetcode.cn/problems/climbing-stairs/) —— `dp[i] = dp[i-1] + dp[i-2]`，斐波那契。
- [最大子数组和](https://leetcode.cn/problems/maximum-subarray/) —— `dp[i] = max(dp[i-1] + nums[i], nums[i])`。
- [最长递增子序列](https://leetcode.cn/problems/longest-increasing-subsequence/) —— `dp[i] = max(dp[j] + 1)` for all j < i where nums[j] < nums[i]。

**常见错误**：
1. dp 数组定义错了——`dp[i]` 到底表示什么？是以 i 结尾还是前 i 个？定义不清，转移方程全错。
2. base case 漏了——`dp[0]` 的值是什么？先想清楚最简单的子问题。
3. 遍历顺序反了——有些 DP（背包问题）要从后往前，否则会重复使用。

### 3.4 贪心算法——当机立断

> "当机立断。"

**核心矛盾**：每一步都有多个选择，全局最优往往需要回溯/DP 来保证。但有些问题中，**局部最优就等于全局最优**——这时候不需要瞻前顾后，当机立断即可。

**原理**：贪心的关键不是"贪"，而是"证明局部最优能推出全局最优"。不能证明的贪心是瞎贪。能证明的贪心是"当机立断"——正确的时机做正确的决定。

**验证贪心是否正确的三步**：
1. 最优子结构：全局最优包含局部最优
2. 贪心选择性质：局部最优选择不会影响全局最优
3. 反例测试：能不能构造一个反例让贪心失败？

**经典例题**：
- [分发饼干](https://leetcode.cn/problems/assign-cookies/) —— 最小饼干满足最小胃口，双指针。
- [跳跃游戏 II](https://leetcode.cn/problems/jump-game-ii/) —— 每步跳到能到达的最远位置。
- [无重叠区间](https://leetcode.cn/problems/non-overlapping-intervals/) —— 按结束时间排序，每次选最早结束的。

**常见错误**：
1. 没证明贪心正确就写——赌贪心，赌输了 WA。
2. 排序后贪心和原顺序贪心搞混——有些题排序后会丢失原始顺序信息。
3. 贪心不适用的题硬套——比如 0/1 背包，贪心选价值密度最高的物品会翻车。

### 3.5 图论基础——联系与发展

> "联系与发展。"

**核心矛盾**：数据不再是一维线性，节点之间形成了复杂的网络关系。**如何在"相互联系"的节点网络中，找到"发展"（遍历、最短路径、连通性）的规律？**

**武器库**：
- **DFS（深度优先）**：一条路走到黑，回头再走别的路。适合：连通性、拓扑排序、找环。
- **BFS（广度优先）**：一层一层扩散。适合：最短路径（无权图）、层序遍历。
- **Dijkstra**：带权图的最短路径，贪心 + 优先队列。
- **并查集**：动态维护连通关系，近乎 O(1) 合并与查询。

**模板：BFS（最短路径）**

```python
from collections import deque

def bfs(start, target):
    queue = deque([start])
    visited = {start}
    step = 0
    while queue:
        size = len(queue)
        for _ in range(size):
            node = queue.popleft()
            if node == target:
                return step
            for neighbor in graph[node]:
                if neighbor not in visited:
                    visited.add(neighbor)
                    queue.append(neighbor)
        step += 1
    return -1
```

**经典例题**：
- [岛屿数量](https://leetcode.cn/problems/number-of-islands/) —— DFS/BFS 遍历二维网格，标记已访问。
- [课程表](https://leetcode.cn/problems/course-schedule/) —— 拓扑排序，检测有向图是否有环。
- [网络延迟时间](https://leetcode.cn/problems/network-delay-time/) —— Dijkstra 单源最短路径。

**常见错误**：
1. BFS 忘记标记 visited——节点重复入队，可能死循环或超时。
2. DFS 递归深度过大——Python 默认递归深度 1000，大规模图用迭代栈。
3. 并查集只压缩 find 不合并 rank——退化成链表，O(n) 查找。

### 4.1 持久战三阶段

> "战争的第一阶段，是敌之战略进攻、我之战略防御的时期。第二个阶段，是敌之战略保守、我之准备反攻的时期。第三个阶段，是我之战略反攻、敌之战略退却的时期。"
>
> ——《论持久战》，1938

学算法不是速决战，是**持久战**。妄想三周刷完三百题然后横扫面试——这叫速胜论，结果必然是挫折和放弃。

**第一阶段：战略防御（第 1-4 周）**

敌人（算法题）强大，我们弱小。这个阶段的任务不是"攻克难题"，而是**建立根据地**。

- 目标：掌握最基础的数据结构——数组、链表、哈希表、栈、队列
- 策略：每天 2-3 道简单题，不求多但求彻底理解"矛盾是什么"
- 关键：防御阶段别碰动态规划！那是速胜论，必然受挫。

**第二阶段：战略相持（第 5-8 周）**

矛盾双方力量对比发生变化。你已经有了一定的基础，可以开始攻坚。

- 目标：掌握核心算法——二分查找、回溯、动态规划入门、BFS/DFS
- 策略：每天 1-2 道中等题，每个算法至少做 5 道变体
- 关键：相持阶段最考验耐心。不要因为一道 DP 题卡了三天就灰心——持久战的核心是坚持。

**第三阶段：战略反攻（第 9-12 周）**

你由弱转强，敌人由强转弱。

- 目标：综合运用、攻克困难题、图论进阶
- 策略：每天 1 道困难题 + 总结归纳，形成自己的方法论体系
- 关键：反攻不是"胡乱冲锋"，要带着"矛盾分析"的框架去解题。每道题先找矛盾，再选武器。
